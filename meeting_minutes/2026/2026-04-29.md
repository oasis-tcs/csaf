#### PR [#1427](https://github.com/oasis-tcs/csaf/pull/1427) - Registry for Vulnerability ID Systems (RVIDS) and governance standing rule

- Thomas Schmidt presented PR #1427, which originated from a comment by Sonny van Lingen during the public comment period requesting clear guidance for the RVIDS. The PR contains:
  - A description of what RVIDS is and how it works (standardized names for vulnerability ID systems used in the `system_name` / `text` properties, with text patterns to validate entries).
  - A mapping mechanism from currently used values to registered values, with the algorithm describing how mappings are produced.
  - A governance and registration process (under the "Changes" section) which is intended to become a standing rule.
- Sonny van Lingen asked whether new items added to the mapping would bypass a TC vote. Thomas Schmidt clarified that only new mapping entries (when the ID system is already in RVIDS, evidence exists in real-world files, and editors confirm no negative impact on other entries) are added by the change controller and approved by editors. New ID systems still require a TC vote.
- Sergii Demianchuk asked whether RVIDS is a response to the EU Cyber Resilience Act. Thomas Schmidt explained the motivation is broader: CVE remains a first-class citizen, but many vendor-specific IDs (e.g. Cisco bug IDs, Red Hat Bugzilla IDs, GitHub issues) are named differently across the supply chain. RVIDS provides authoritative system names to correlate advisories that refer to the same vulnerability. Discussion followed about how this helps tracking across the supply chain (e.g. Log4Shell, where vendors initially used internal bug IDs before a CVE was assigned).
- Sonny van Lingen noted that mixing this with SBOM linkage goes beyond the intended scope; Thomas Schmidt confirmed this is mainly about identifying the same vulnerability via standardized system names (especially important for VEX where a CVE may not exist).
- Sergii Demianchuk pointed out that the link in the registry currently redirects to raw GitHub. Thomas Schmidt confirmed the long-term plan is for the registry to be served directly under `registry.csaf.io`.
- Sonny van Lingen asked whether EUVD should be voted in as a registered system to maintain procedural hygiene. Thomas Schmidt agreed to vote on the existing entries.

A motion was presented by Thomas Schmidt to accept PR #1427 into the CSAF 2.1 repository, with the caveat of not adopting the standing rule suggested in lines 65 and following (which would be voted separately). The motion was seconded by Sergii Demianchuk (also seconded by Christoph Plutte). No objections, discussions or concerns were raised. The motion carries.

A motion was presented by Thomas Schmidt to accept EUVD as a registered system into RVIDS as stated in the registry. The motion was seconded by Christoph Plutte. No objections, discussions or concerns were raised. The motion carries.

A motion was presented by Thomas Schmidt to accept the OASIS Open CSAF TC GitHub Issues as a registered system into RVIDS as stated in the registry. The motion was seconded by Christoph Plutte. No objections, discussions or concerns were raised. The motion carries.

#### Issue [#1380](https://github.com/oasis-tcs/csaf/issues/1380) - Test for `discovery_date`

- Thomas Schmidt presented Issue #1380, which suggests an additional test ensuring the `discovery_date` is not later than the `disclosure_date` and not later than the newest date in the revision history.
- Sonny van Lingen raised a discussion about multi-vendor CVD (coordinated vulnerability disclosure) scenarios where vendor B's discovery date could be after vendor A's disclosure date.
- Christoph Plutte argued that, per the field definitions, `discovery_date` refers to the original discovery of the vulnerability in its narrower scope (e.g. in the affected component library), not a downstream consumer's discovery. Defining the temporal dependencies this way provides clarity.
- Thomas Schmidt rephrased the motion as a call to action: TC members are requested to review Issue #1380, think about edge cases, and post them on the issue so a better-informed discussion can take place. The vote was deferred.
- Sonny van Lingen suggested looking at real-world data points from large document collections.

#### Issue [#1348](https://github.com/oasis-tcs/csaf/issues/1348) - Remove leap second conversion rule

- Thomas Schmidt presented Issue #1348, which proposes to remove the leap second conversion rule as it is considered non-significant.
- Thomas Schmidt argued the leap second conversion rule is actively used by implementers as guidance for handling leap seconds (which are not well supported by many libraries), both for conversion and for handling such situations in CSAF 2.0.
- A motion was presented by Thomas Schmidt to reject the suggestion in Issue #1348 and keep the leap second conversion rules in CSAF 2.1. The motion was seconded by Sergii Demianchuk, who recalled the original discussion in April 2025 at the in-person meeting and reaffirmed support. No objections, discussions or concerns were raised. The motion carries.
