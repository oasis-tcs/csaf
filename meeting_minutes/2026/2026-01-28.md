#### Standing rule "Known Issue Rule" (Full Majority Vote)

- Thomas Schmidt presented the proposed standing rule (originally circulated by email): rather than holding two motions whenever a fix is brought into a new version (one to bring the fix in, another to add the same item to the known-issues list of previous versions), the editors would be granted the right to add the item to the known-issues list of prior versions if applicable, without a separate motion.
- Justin Murphy stated the proposal made sense to him; no other questions were raised.
- Sergii Demianchuk asked for clarification whether a voting member without write access could make such additions. Thomas Schmidt clarified that only those with write access can make the change directly, but any TC member can open a pull request which editors can then merge using the standing rule.
- A motion was presented by Thomas Schmidt to introduce the following standing rule, called the **Known Issue Rule**: *"The editors can add any item fixed in a version of the standard to the list of known issues for all previous versions if applicable without an extra motion. Also, items that are acknowledged as a known issue in one version can be added to other versions of the standard if applicable. The process of having at least one approval and no rejection from another member with write access remains in place."*
- The motion was seconded (per Justin Murphy's confirmation: Sonny van Lingen, Sergii Demianchuk and Stefan Hagen had each voted/seconded via chat early). No objections, discussions or concerns were raised. A **Full Majority Vote** was held; voting was conducted via the meeting chat. The following Voting Members and Co-Chairs present voted in favor:

  - Christian Banse
  - Christoph Plutte
  - Feng Cao
  - Jessie Vaught
  - Justin Murphy
  - Sergii Demianchuk
  - Sonny van Lingen
  - Stefan Hagen
  - Thomas Schmidt

  The Full Majority Vote was successful. The Known Issue Rule standing rule was adopted.

#### Issue [#1218](https://github.com/oasis-tcs/csaf/issues/1218) - Distribution ordering of `changes.csv` / ROLIE feed

- Thomas Schmidt introduced the issue: both the ROLIE-based and file-based distributions rely on sorting by `current_release_date`, but some providers insert documents back-dated to historical positions in the list, causing iterating downloaders to miss them.
- Two options were discussed:
  1. Change the rules for `changes.csv` and the ROLIE feed so that entries must always be inserted at the top (and not re-sorted), allowing consumers to reliably read from the most recent entry downwards.
  2. Add an additional "uploaded" timestamp to `changes.csv` and the ROLIE feed so consumers can sort by upload time.
- Christoph Plutte (Ericsson PSIRT) discussed the implications of option 1 — the date would become an indicator rather than the sort key — and noted there is no fully beautiful solution because the underlying situation (non-compliant providers) is itself problematic.
- Stefan Hagen observed option 2 would effectively mimic file-system update/creation dates.
- No motion was set. Thomas Schmidt asked TC members to consider the options, comment on the issue, and committed to providing a concrete suggestion in the upcoming days.

#### Issue [#1177](https://github.com/oasis-tcs/csaf/issues/1177) - Dependency path concept

- Christoph Plutte (Ericsson PSIRT) presented a simplified version of his original proposal, jointly worked out with Thomas Schmidt. The idea is to introduce a new "dependency path" object in the product tree that can describe a chain from the root of a dependency tree (the main product) through multiple intermediate nodes down to a leaf component, addressing each occurrence of a component separately - especially useful for cloud-native and complex supply-chain products where the same library can appear in multiple container images or microservices with different attack surfaces.
- Sergii Demianchuk asked for time to read the proposal and asked what concrete issues it solves. Christoph explained that pure tuple relationships cannot describe these multi-instance cloud-native situations without ambiguity. Sergii agreed that more precise specification is better.
- Tobias Limmer (Siemens) supported the proposal in general, stating Siemens sees the issue in complex supply chains and the feature is needed, while noting he still needs to read all comments before a final call.
- Sonny van Lingen (no opposition) requested a final consolidated proposal in the issue so that subsequent acceptance is clearer.
- Stefan Hagen voiced support and suggested moving forward with an implementation PR so the TC can decide on the basis of a concrete change.
- Christian Banse asked whether the new concept is intended to replace the existing relationship object. Thomas Schmidt confirmed it would replace relationships (avoiding two parallel ways to express the same thing). Christian indicated he supports the new approach as cleaner.
- Outcome: No motion at this meeting. Thomas Schmidt to provide a pull request; TC members to review and provide feedback ahead of the next meeting / via the mailing list before a motion is set.

#### Issue [#1044](https://github.com/oasis-tcs/csaf/issues/1044) - GitHub Copilot capabilities in the repository

- Outcome: No motion. Raised as awareness for the TC.
