# CSAF v2.1 Prose â€” Edit Workspace

This folder is the authoring workspace for the CSAF v2.1 specification.
Sources are split across 228 Markdown files in `src/`, one per section or subsection.
The `makefile` drives the full build pipeline via [nide](https://pypi.org/project/nide/), pandoc, and typst.

## Folder layout

| Path       | Contents                                                              |
|:-----------|:----------------------------------------------------------------------|
| `src/`     | One Markdown file per section; the canonical authoring surface        |
| `etc/`     | Pipeline configuration, templates, mapping tables, and linting config |
| `bin/`     | Utility scripts for editorial maintenance tasks                       |
| `makefile` | All build, lint, diff, and release targets                            |

### `etc/` files

| File                            | Purpose                                                           | Edit?        |
|:--------------------------------|:------------------------------------------------------------------|:-------------|
| `nide.yaml`                     | nide pipeline configuration (channels, paths, options)            | yes          |
| `bind.txt`                      | Assembly order â€” lists source files in document sequence          | yes          |
| `csaf.typ`                      | Typst document template (CSAF-specific layout)                    | yes          |
| `oasis.typ`                     | Typst OASIS branding and style definitions                        | yes          |
| `section-display-to-label.json` | Maps display numbers to section labels                            | **yes**      |
| `section-label-to-display.json` | Inverse of the above; derived by `nide harvest`                   | no â€” derived |
| `section-display-to-text.json`  | Maps display numbers to section titles; derived by `nide harvest` | no â€” derived |
| `example-global-to-local.json`  | Maps global example IDs to per-section local IDs                  | **yes**      |
| `example-local-to-global.json`  | Inverse of the above; derived by `nide harvest`                   | no â€” derived |
| `markdownlint.json`             | markdownlint rule configuration                                   | yes          |
| `vale.ini`                      | Vale prose linter configuration                                   | yes          |
| `tidy-config.txt`               | HTML tidy settings applied during the HTML build                  | yes          |

**Mapping file discipline:** only the two files marked **yes** above are hand-edited.
Their inverses are regenerated by `nide harvest` (the `make sections` target) every time a build runs.
Never edit the derived files directly and expect the changes to be preserved across builds â€” your changes will be overwritten on the next build.

## The authoring loop

    # 1. Edit one or more source files
    $EDITOR src/some-section.md

    # 2. Build HTML to review locally
    make html

    # 3. Check document completeness against editorial rules
    make quality       # per-rule PASS/FAIL/NA report

    # 4. When ready for a round-trip review, build all channels
    make release

    # 5. Verify nothing unintended changed
    make diff          # prose diff of HTML against VCS HEAD
    make diff-repro    # reproducible diff of PDF against VCS HEAD

    # 6. Check VCS status, then commit
    make status

## Using `nide quality` directly

The make target runs `nide quality` with the rules file configured in `etc/nide.yaml`.
For finer control, run it directly after `make gfm-plus` (which writes `build/ir.json`):

    # default: reads quality.rules from etc/nide.yaml
    nide quality

    # explicit rules file
    nide quality --rules etc/rules/spec.rules.yaml

    # machine-readable output (for CI or scripting)
    nide quality --json | jq '.[] | select(.result == "FAIL")'

    # promote WARN to BLOCKER (strict mode)
    nide quality --strict

The rules file `etc/rules/spec.rules.yaml` extends `etc/rules/oasis.rules.yaml`
(the shared OASIS publication-assurance baseline) and adds CSAF-specific checks.
`oasis.rules.yaml` comes from the `OASIS-Docs/publication-assurance` repository â€”
copy it from there; do not edit it locally.

## Using `nide diff` directly

The make targets cover the common cases.
For finer control, run `nide diff` directly (VCS version is extracted automatically):

    # prose mode â€” readable word-level diff
    nide diff --mode prose ../share/csaf-v2.1-draft.html

    # repro mode â€” suppress build-timestamp noise
    nide diff --mode repro ../share/csaf-v2.1-draft.pdf

    # meta mode â€” show only XMP / header metadata changes
    nide diff --mode meta ../share/csaf-v2.1-draft.pdf

    # prose diff with XMP block appended
    nide diff --mode prose+xmp ../share/csaf-v2.1-draft.html

    # full mode â€” content + blobs + timestamps + XMP
    nide diff --mode full ../share/csaf-v2.1-draft.html

All mode names and modifier names accept unambiguous prefixes (`nide diff --mode pr` is equivalent to `--mode prose`).
